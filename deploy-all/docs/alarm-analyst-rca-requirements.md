# Alarm Analyst 根因分析流程需求文档

## 1. 文档目的

本文档定义 `alarm-analyst` 技能在 QwenPaw / Portal 场景下的**完整根因分析流程**、输入输出约束、跨系统调用顺序、失败回退策略与验收标准。

目标不是写一个“可选参考流程”，而是明确：

- 哪些步骤是**必须执行**
- 哪些字段是**必须携带**
- 哪些接口是**必须调用**
- 哪些结果必须**一起交给 AI** 做最终 RCA

本文档适用于以下实现面：

- Portal 右上角铃铛告警点击进入故障处置员
- `alarm-analyst` 技能脚本
- `portal_backend` / `fault_scenario_service`
- `real-alarm`、`veops-cmdb`、指标接口的协作链路

---

## 2. 业务背景

当前场景不是简单“看一条告警”，而是围绕某条具体活动告警完成结构化 RCA：

1. 用户在 Portal 中看到活动告警
2. 用户点击某条告警
3. 系统把告警上下文发送给故障处置员数字员工
4. 系统围绕该告警对应的 `resId / CI ID` 完成：
   - 根资源确认
   - CMDB 拓扑扩散
   - 关联资源告警采集
   - 根资源指标采集
   - AI 根因分析
5. 最终向用户输出结构化 RCA 结果，并为后续处置/闭环提供依据

---

## 3. 术语定义

| 术语 | 说明 |
|------|------|
| `resId` | 资源 ID，等同于 CMDB 中的 `CI ID` |
| 根资源 | 当前告警直接对应的资源，即由告警携带的 `resId` 指向的资源 |
| 拓扑关联资源 | 在 CMDB 中与根资源存在关系的全部资源节点 |
| 当前窗口 | 以当前告警发生时间为中心，默认前后 10 分钟 |
| 环比窗口 | AI 可指定的对比窗口；若未指定，默认使用前一等长窗口 |
| RCA | Root Cause Analysis，根因分析 |

---

## 4. 输入要求

`alarm-analyst` 执行时，最少应接收以下字段：

| 字段 | 是否必填 | 说明 |
|------|----------|------|
| `resId` | 是 | 当前告警对应资源 ID / CI ID |
| `eventTime` | 是 | 当前告警发生时间 |
| `alarmTitle` | 建议 | 当前告警标题 |
| `deviceName` | 建议 | 当前告警设备名称 |
| `manageIp` | 建议 | 当前告警管理 IP |

### 4.1 Portal 传递要求

Portal 点击铃铛告警时，发给故障处置员的文本中必须至少补充：

```text
资源 ID（CI ID）：<resId>
告警时间：<eventTime>
```

原因：

- `resId` 是后续根资源、拓扑扩散、告警 fan-out、指标采集的唯一锚点
- `eventTime` 是“当前窗口前后 10 分钟”查询的时间锚点

如果没有这两个字段，系统只能退回脚手架结果，不能执行完整 RCA。

---

## 5. 完整逻辑链

以下顺序为**必须遵循的执行链**：

### 步骤 1：确认根资源

以当前告警的 `resId` 为输入，调用 `veops-cmdb` 查询根资源详情。

目标：

- 明确根资源是否存在
- 明确根资源名称
- 明确根资源 `ciType`

这一阶段的结论是后续指标采集的唯一依据。

### 步骤 2：查询根资源拓扑关系

仍然使用同一个 `resId` 作为根节点，调用 `veops-cmdb` 查询 CMDB 拓扑关系。

目标：

- 获取根资源关联的全部拓扑节点
- 获取关系边两端的资源 ID
- 获取嵌套资源节点中的资源 ID

这一阶段不能只关注可展示的树图，而必须关注“可用于 fan-out 查询告警的资源 ID 集合”。

### 步骤 3：收集全部相关资源 ID

从拓扑关系返回中提取**全部**相关资源 ID。

必须覆盖：

- 根资源本身的 `resId`
- 拓扑节点中的 `_id / id / ci_id / ciId`
- 关系边中的 `src_ci_id / dst_ci_id`
- 嵌套 `parent / child / source / target` 里的资源对象 ID

不能只取：

- 单个节点 ID
- 单条关系的一端
- 仅根资源本身

输出要求：

- 形成一个去重后的 `resourceIds` 集合
- 根资源 ID 排在第一位

### 步骤 4：根据全部相关资源 ID 查询告警

这一步是**必须执行**的，且必须在“收集完成全部相关资源 ID”之后执行。

对于集合中的每一个资源 ID，逐个调用实时告警接口：

```http
POST /resource/realalarm/list
```

调用规则：

- 当前资源 ID 必须放进请求体里的 `neId`
- 查询不是只针对根资源，而是针对拓扑中的全部相关资源

### 步骤 5：查询当前窗口告警

默认当前窗口定义为：

- `eventTime - 10 分钟`
- `eventTime + 10 分钟`

即：

```text
recentBeginTime = eventTime - 10min
recentEndTime   = eventTime + 10min
```

这一步的目的不是看“历史趋势”，而是看：

- 当前告警附近时间点，拓扑链路上有哪些伴随告警
- 同一时间段是否有扩散、级联、伴随异常

### 步骤 6：查询环比窗口告警

环比窗口允许两种方式：

1. AI 主动指定时间范围  
2. 未指定时，默认使用前一等长窗口

默认规则：

```text
previousEndTime   = recentBeginTime
previousBeginTime = previousEndTime - (recentEndTime - recentBeginTime)
```

目的：

- 判断关联告警是否在当前窗口显著增加
- 判断问题是新出现、扩散中、还是已有收敛趋势

### 步骤 7：基于根资源 `ciType` 查询指标定义

完成告警 fan-out 后，再根据**根资源**的 `ciType` 查询指标定义：

```http
POST /resource/resource/threshold/getMetricDefinitions
```

说明：

- 指标定义接口路径是双 `resource`
- 指标类型来源于根资源 `ciType`
- 不能通过拓扑里其他节点类型替代根资源类型

### 步骤 8：筛选关联指标

指标定义返回后，AI 需要从候选指标中筛选与当前故障最相关的指标。

典型 MySQL 场景重点关注：

- 锁等待
- 死锁
- 慢 SQL
- 长事务
- InnoDB 行锁时间
- 活跃线程
- 连接压力

### 步骤 9：遍历查询指标值

对 AI 选中的指标逐个调用：

```http
POST /resource/pm/getMetricData
```

说明：

- 指标值接口路径是单 `resource`
- `mulRes[].resId` 必须使用根资源 `resId`
- `queryKeys` 每次只传一个 `metricCode`

### 步骤 10：把所有上下文一起交给 AI 做 RCA

最终交给 AI 的 RCA 输入必须包含以下全部内容：

1. 根资源详情
2. 根资源 `ciType`
3. CMDB 拓扑关系
4. 拓扑全部相关资源 ID
5. 当前窗口关联告警
6. 环比窗口关联告警
7. 当前窗口与环比窗口的比较结果
8. 根资源的候选指标定义
9. 根资源关键指标的指标值

不能只给：

- 一组指标
- 一条告警
- 或者只有拓扑摘要

## 6. 实现要求

### 6.1 `veops-cmdb`

- 负责根资源详情查询
- 负责拓扑关系查询
- 固定读取自身技能目录下的 `.env`
- 查询接口优先匿名访问，401/403 再登录
- Python 网络失败时允许回退到系统 `curl`

### 6.2 `real-alarm`

- 负责对拓扑资源 ID 逐个 fan-out 查告警
- 请求体中必须使用 `neId = resourceId`
- Python 网络失败时允许回退到系统 `curl`

### 6.3 `alarm-analyst`

- 负责组织完整链路
- 不能只查指标
- 不能只查根资源告警
- 必须显式完成：
  - 根资源确认
  - 拓扑扩散
  - 告警 fan-out
  - 指标定义
  - 指标值采集
  - RCA 结果整合

### 6.4 qwenpaw / portal 运行入口

- 实际运行入口必须调用 `alarm-analyst` 聚合脚本
- 不能继续返回旧的“脚手架结果”
- Portal 点击告警时必须把 `resId` 和 `eventTime` 注入消息

---

## 7. 错误处理要求

### 7.1 根资源缺失

如果 `resId` 对应资源查不到：

- 终止后续拓扑与指标分析
- 明确返回：根资源不存在或无法确认

### 7.2 拓扑为空

如果拓扑关系为空：

- 仍然必须保留根资源本身
- 后续至少要对根资源查询告警和指标

### 7.3 告警接口失败

如果某个资源的告警查询失败：

- 明确记录失败资源 ID
- 已成功资源的告警仍然保留
- 不能假装“全量告警都已拿到”

### 7.4 指标接口失败

- `getMetricDefinitions` 失败时，可回退 mock 指标定义
- `getMetricData` 失败时，可回退 mock 指标值
- 但必须在结果中标记为回退

---

## 8. 输出要求

最终返回给 AI 或用户的结果至少包含以下结构块：

1. 当前告警
2. 根资源详情
3. CMDB 拓扑
4. 当前窗口关联告警
5. 环比窗口关联告警
6. 环比比较结果
7. 指标分析
8. 初步关系判断 / 根因候选

---

## 9. 验收标准

满足以下条件才算完成：

1. 给定一个告警 `resId` 和 `eventTime`
2. 系统能够从 CMDB 查到根资源详情
3. 系统能够从拓扑中提取全部相关资源 ID
4. 系统能够对这些资源 ID 逐个调用实时告警接口
5. 当前窗口默认是 `eventTime ± 10 分钟`
6. 环比窗口可自动生成，也可由 AI 指定
7. 系统能够依据根资源 `ciType` 调指标定义接口
8. 系统能够对筛选出的指标逐个调指标值接口
9. 最终输出同时包含：
   - 拓扑告警信息
   - 指标数据
   - RCA 所需的综合上下文

---

## 10. 明确禁止事项

以下行为视为不符合需求：

1. 只查根资源自己的告警，不查拓扑关联资源
2. 只取拓扑里一个资源 ID，不收集全部相关资源 ID
3. 不把资源 ID 放到 `neId`
4. 不按告警发生时间前后 10 分钟查当前窗口
5. 没有环比窗口或环比能力
6. 不查根资源 `ciType` 就直接猜指标类型
7. 只查指标，不查拓扑 fan-out 告警
8. 只查拓扑 fan-out 告警，不查指标
9. Portal / qwenpaw 实际运行入口仍然走脚手架结果而不是完整链路
